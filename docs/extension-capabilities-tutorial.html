How to Use AI Work Assistant

A guided tour of everything the extension can currently do: local todos, Azure DevOps sync, Outlook email triage, planner output, repo intelligence, and approval-gated actions.

Before You Start

  • Start the backend with ./scripts/dev-agent.sh.
  • Launch the extension with F5 or install the packaged VSIX.
  • Open the AI Work Assistant icon in the VS Code activity bar.

Safe by Default

  • The assistant reads local data and syncs external context only when commanded.
  • File writes require explicit approval.
  • Outlook creates drafts only; it never sends email.

Main Workflow

  1. Sync tickets and emails.
  2. Review generated todos.
  3. Run Plan My Day.
  4. Work the top item and update status.

1. Open the Assistant Sidebar

  1. Open VS Code.
  2. Start the backend: ./scripts/dev-agent.sh.
  3. Press F5 to launch the Extension Development Host, or install the VSIX.
  4. Click the AI Work Assistant activity bar icon.

The sidebar talks to the local FastAPI backend at http://127.0.0.1:8765 by default.

2. Manage Local Todos

The sidebar includes a todo form and a todo list.

Create a Todo

  1. Enter a title.
  2. Add an optional description.
  3. Choose a priority.
  4. Click Add.

Update Status

Use the status dropdown on each todo to move it through:

  • pending
  • in_progress
  • done
  • canceled

Refresh

Click Refresh to reload todos from SQLite.

3. Sync Azure DevOps Tickets

The extension can read assigned Azure DevOps work items and sync them into local todos. Organization and project are settings, not hardcoded.

Configure Settings

  1. Open VS Code Settings.
  2. Search for AI Work Assistant.
  3. Set aiWorkAssistant.azureDevOps.organization.
  4. Set aiWorkAssistant.azureDevOps.project.

Run Sync

Command Palette → Assistant: Sync DevOps Tickets

The first time, the extension asks for a Personal Access Token. It stores the PAT in VS Code SecretStorage and sends it only to the local backend for that request.

What You Get

  • Assigned work items become local todos.
  • Todos are linked to Azure DevOps IDs.
  • Tickets are categorized as blocked, urgent, due soon, in progress, or normal.

4. Update the Current Azure DevOps Ticket

Use this after you make progress on a linked ticket.

Command Palette → Assistant: Update Current Ticket
  1. Select an Azure DevOps-linked todo.
  2. Choose the target ticket state.
  3. The backend updates Azure DevOps using your stored PAT.
  4. The matching local todo status is updated.

5. Sync Outlook Unread Emails

Outlook integration uses Microsoft Graph. It reads unread inbox messages and generates local todos from actionable emails. 

Command Palette → Assistant: Sync Outlook Emails

The first time, enter a Microsoft Graph access token with appropriate mail permissions.

Email Categories

Category Meaning
pipeline_failure Build, deployment, release, or pipeline failure detected.
needs_response Email likely needs a written reply.
needs_action Email asks you to review, approve, or do something.
meeting_context Agenda, recap, or meeting-related context.
informational Useful but not urgent.

6. Pipeline Failure Handling

Pipeline failure emails get special treatment.

  • They become urgent todos.
  • The assistant tries to identify the repo.
  • The assistant tries to identify the pipeline.
  • The todo description includes a recommended next action.

Typical next action:

Open the failed pipeline run, inspect the first failing task,
check recent commits, and verify environment variables.

7. Create Outlook Draft Responses

The app can create draft replies, but it never sends email.

Command Palette → Assistant: Create Outlook Draft Response
  1. Select an Outlook-linked todo.
  2. Enter draft response text.
  3. Review the approval prompt.
  4. Approve only if you want a draft created in Outlook.

This is an outbound action, so it is approval-gated. The backend requires an approval header before calling Microsoft Graph to create the draft.

8. Plan Your Day

The planner reads local todos from tickets, emails, blocked items, overdue work, and repo context.

Command Palette → Assistant: Plan My Day

Planner Output

  • Prioritized work list.
  • Recommended next task.
  • Estimated effort.
  • Explainable scoring reasons.
  • Suggested Azure DevOps ticket updates.

Optimization Goals

  • Prioritize blockers.
  • Prioritize urgent production issues.
  • Reduce context switching by favoring in-progress work.
  • Maximize momentum with concrete next actions.

9. Use Repo Intelligence

Repo intelligence scans the configured local repository root. It detects project type, important files, git metadata, deployment hints, and validation commands.

Useful Local API Calls

curl http://127.0.0.1:8765/repo/context
curl http://127.0.0.1:8765/repo/architecture
curl http://127.0.0.1:8765/repo/deployment-flow

What It Detects

  • Databricks Asset Bundles and databricks.yml.
  • Python projects.
  • SQL files.
  • dashboard-style repos.
  • CI/CD repos.
  • TypeScript projects.

10. Run Approved Repo Commands

These commands are available through the backend and require explicit approval.

Command ID Runs
databricks_bundle_validate databricks bundle validate
pytest pytest
ruff ruff check .
mypy mypy .

Example

curl -X POST http://127.0.0.1:8765/repo/commands/run \
  -H "content-type: application/json" \
  -H "X-Approval-Decision: approved" \
  -d '{"command": "pytest"}'

11. Approval-Gated File Updates

The backend can update files inside the configured repo root, but only with explicit approval. This is the safety boundary for future edit/refactor/code-generation features. 

curl -X POST http://127.0.0.1:8765/repo/files/update \
  -H "content-type: application/json" \
  -H "X-Approval-Decision: approved" \
  -d '{
    "relative_path": "notes/example.md",
    "content": "Approved local file update"
  }'

Paths are constrained to the repository root. Attempts to write outside the repo are rejected.

12. What the App Does Not Do Yet

  • It does not run a real LLM workflow yet.
  • It does not auto-send email.
  • It does not auto-modify files without approval.
  • It does not deploy Databricks bundles automatically.
  • It does not hardcode your organization, project, or repo names.

13. Suggested Daily Workflow

  1. Start backend: ./scripts/dev-agent.sh.
  2. Open VS Code assistant sidebar.
  3. Run Assistant: Sync DevOps Tickets.
  4. Run Assistant: Sync Outlook Emails.
  5. Review generated todos.
  6. Run Assistant: Plan My Day.
  7. Work the recommended next task.
  8. Update the linked ticket when progress changes.
  9. Run approved repo validation commands before committing.
